Creating a Virtual Environment:

When creating Django projects you need to create a virtual Environment
This ensures there are no clashing dependancies between projects
They keep your project isolated.
Allows the computer to have multiple versions of Django and Python installed on the machine without them clashing with eachother

So if you switch between packages, libraries or versions it does not affect other projects because you are only making the changes
within the given virtual Environment

For example, if you are working on an open source project that uses django 1.5, but you have django 1.9 installed, you will get loads
of errors when you try and run the code. And if you try to downgrade to django 1.5, then it may work but all your other projects that
were built using django 1.9 will now no longer work.

Creating a virtual environment allows isolation of dependancies, enabling the installation and management of project specific requirements
without impacting the global python or Django installation on your machine. Using a virtual env gives you complete control of the packages
and tools used for a given project and makes sure that they do not clash with other projects

You essentially download a local installation of Python and Django within the virtual environment that no longer interferes with anything
outside of that environement

A migration in Django is a way of applying and tracking changes to your database schema.
Think of migrations as the Git for your database structure.
It essentially tracks the changes, ensures they are safe, and then commits them to the database

When you create a migration, Django will create a migrations folder, that will create a new .py file every time a migration is made.
Inside these py files will detail the changes made to the database during that migration

Even if we declare a migration using 'makemigrations', this only documents/tracks the changes.

What is the rest framework?
'''
- The REST api framework is a Django python toolkit designed to build web APIs on top of Django.
- It is known as the Django Rest Framework (DRF)

Part of the DRF is the 'request' object. This wraps Django HttpRequest and adds request.data, automates JSON parsing and authenticates hooks
Hooks are the behaviour and flow of an API request. Without this alot of the python data would need manual parsing to JSON

DrinkSerializer is a Django seriliazer class that serializes models which are objects that go within the database
Serialization is needed because Django works with internal data that is not compatible with the outside world (HTTP/JSON)

Django works with model instances, query sets (multiple objects), and Python objects
Outside of Django the client (browser) uses JSON, Dictionaries, Lists etc 

JSON does not understand models, foreign keys, meta data etc that come with Django query sets from an SQL database

'''
What 'serializer = DrinkSerializer(drinks, many=True)' actually does?
'''

- States which model fields are allowed to leave the server, which columns in the database are we alloweed to use
- Converts the data from model fields to plain python, JSON compatible and safe to send over HTTP 
- We also need it to deserialize data from JSON->Python and then from Python->Django model fields

Database -> Django Model Fields -> Serializer -> Plain Python -> JSON -> Client (webserver) -> JSON -> Plain Python -> Serialzier -> Django Model Fields -> Database

Serilaizers are essential in APIs because the offer:
- Security
- Validation of data
- Explicit API contracts
- Control over exposed data
- Correct JSON output
- Bidirectional conversion (GET/POST)
